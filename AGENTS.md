# Agent 编码规则

本文档定义了本项目 AI 辅助开发时代码编写必须遵循的规范，在通用 Python/PEP 8 规范基础上补充了项目特有的要求。

## 一、注释要求

### 1.1 必加注释位置

新增或修改代码时，**每个定义上方都必须添加中文注释**：
- 类
- 方法
- 函数
- 接口
- 类型
- 常量

### 1.2 注释内容要求

注释需要说明：
- 该定义的**职责**
- 该定义的**核心用途和属性**
- 关键**约束条件**

❌ 禁止只重复名称本身（例如 `# 初始化函数` 这种废话注释）。
✅ 注释应该提供额外信息，帮助读者理解为什么这么写，而不是写了什么。

### 1.3 关键行为必须说明

如果定义涉及以下场景，注释中**必须写清楚**核心规则和边界条件：
- 业务规则
- 执行链路
- 比对逻辑
- Trace 处理
- 超时配置
- 并发隔离

### 1.4 修改代码时补齐注释

修改已有定义时，如果原定义缺少中文注释，**需要在本次修改中补齐**：
- 补齐的注释格式应与本规范保持一致
- 如果无法确定原定义的完整意图，应主动询问需求方或代码作者，确认其意图。

## 二、异步任务处理规范

后台异步任务、延迟任务、补偿任务和重试任务：

1. ❌ 禁止只有 happy path 或裸 `return`
2. ✅ 必须明确处理**成功、失败、异常**三类收口路径
3. ✅ 失败或异常时必须：
   - 更新相关业务状态
   - 写入错误信息
   - 更新完成时间
4. 目的：避免页面或数据库记录**长期停留在处理中状态**

## 三、提交代码前检查清单

提交代码前，请逐条确认：

### 注释规范

- [ ] 每个类都有中文 docstring，说明职责和依赖
- [ ] 每个函数/方法都有中文注释，说明功能、参数、返回值、异常
- [ ] 每个常量、类型别名、枚举值都有注释说明含义
- [ ] 注释说明了业务规则、边界条件、关键行为
- [ ] 注释不是简单重复名称，而是提供额外信息

### 关键行为注释

- [ ] 业务规则代码有规则说明
- [ ] 比对逻辑代码有比对规则说明
- [ ] 超时配置代码有超时值和超时后行为说明
