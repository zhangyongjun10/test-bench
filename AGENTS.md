# Agent 编码规则

本文档定义了本项目 AI 辅助开发时代码编写必须遵循的规范，在通用 Python/PEP 8 规范基础上补充了项目特有的要求。

## 一、注释要求

### 1.1 必加注释位置

新增或修改代码时，**每个定义上方都必须添加中文注释**：
- 类
- 方法
- 函数
- 接口
- 类型
- 常量

### 1.2 注释内容要求

注释需要说明：
- 该定义的**职责**
- 该定义的**核心用途和属性**
- 关键**约束条件**

❌ 禁止只重复名称本身，例如 `# 初始化函数` 这种废话注释。  
✅ 注释应该提供额外信息，帮助读者理解为什么这么写，而不是写了什么。

### 1.3 关键行为必须说明

如果定义涉及以下场景，注释中**必须写清楚**核心规则和边界条件：
- 业务规则
- 执行链路
- 比对逻辑
- Trace 处理
- 超时配置
- 并发隔离

### 1.4 修改代码时补齐注释

修改已有定义时，如果原定义缺少中文注释，**需要在本次修改中补齐**：
- 补齐的注释格式应与本规范保持一致
- 如果无法确定原定义的完整意图，应主动询问需求方或代码作者，确认其意图

## 二、异步任务处理规范

后台异步任务、延迟任务、补偿任务和重试任务：

1. ❌ 禁止只有 happy path 或裸 `return`
2. ✅ 必须明确处理**成功、失败、异常**三类收口路径
3. ✅ 失败或异常时必须：
   - 更新相关业务状态
   - 写入错误信息
   - 更新完成时间
4. 目的：避免页面或数据库记录**长期停留在处理中状态**

## 三、提交与变更记录规范

### 3.1 CHANGELOG 更新要求

每次提交代码前，**必须同步更新 `CHANGELOG`**，至少说明：
- 本次改动的功能点或修复点
- 影响范围
- 重要的兼容性、配置或迁移信息
- 任何可能影响业务逻辑的变更（如新增/删除/修改业务规则）
- 只能新增变更记录，不能删除或修改已有的变更记录

### 3.2 提交前检查要求

如果本次改动已准备提交，但未更新 `CHANGELOG`，则视为**提交流程未完成**，不能直接提交。

## 四、提交代码前检查清单

提交代码前，请逐条确认：

### 注释规范

- [ ] 每个类都有中文 docstring，说明职责和依赖
- [ ] 每个函数/方法都有中文注释，说明功能、参数、返回值、异常
- [ ] 每个常量、类型别名、枚举值都有注释说明含义
- [ ] 注释说明了业务规则、边界条件、关键行为
- [ ] 注释不是简单重复名称，而是提供额外信息

### 关键行为注释

- [ ] 业务规则代码有规则说明
- [ ] 比对逻辑代码有比对规则说明
- [ ] 超时配置代码有超时值和超时后行为说明

### 提交与记录

- [ ] 本次改动已同步更新 `CHANGELOG`
- [ ] `CHANGELOG` 已记录功能点、影响范围和必要的配置或迁移说明

## 五、中文编码与文件写入规范

### 5.1 中文文件禁止使用不安全写回方式

修改包含中文的文件时，严禁使用容易导致乱码或字符丢失的写回方式，特别是：
- PowerShell 管道转字符串后再 `Set-Content`
- 未明确编码的重定向覆盖
- 会把中文写成 `?`、乱码或错误字节序的任何临时脚本

### 5.2 必须使用 UTF-8 安全方式修改中文

修改中文内容时必须优先使用以下方式：
- 优先使用能保留 UTF-8 内容的 patch 方式修改
- 如需使用脚本写回文件，必须明确按 `UTF-8` 读写，不能依赖系统默认编码

### 5.3 修改后必须做中文内容校验

凡是修改含中文的文件，在向用户回报完成前必须再次检查：
- 文件内的中文是否正常可读
- 是否出现 `?`、明显乱码或错误字符替换
- 如果发现中文异常，必须先修复编码问题，不能直接交付或继续在错误文件上追加修改
