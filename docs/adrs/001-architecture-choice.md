# ADR-001: 整体架构选择 - 分层单体

## Context

我们需要为 Agent 验证平台选择整体架构。项目需求：
- 第一期 MVP 需要尽快交付
- 第一期只支持单任务并发，用户是内部测试工程师
- 未来需要扩展到 1000+ 并发和多租户对外服务
- 部署要求一期使用 Docker Compose，二期使用 Kubernetes

可选方案：
1. 单体 Monolith - 所有模块在一个进程
2. 微服务 - 每个领域一个独立服务
3. 分层单体 - 模块分层，但都在一个进程

## Decision

**选择：分层单体（方案 C）**

## Rationale

### 为什么不选纯单体（方案 A）：
- 纯单体模块耦合，后续难维护难扩展
- 我们已知未来需要扩展，从一开始就应该有好的结构

### 为什么不选微服务（方案 B）：
- 对 MVP 来说过度设计
- 开发复杂度高，交付时间会翻倍
- 部署复杂，不符合 Docker Compose 本地部署要求
- 团队负担重，需要处理分布式问题（服务发现、分布式追踪、故障排查）

### 为什么选择分层单体：
- 模块按 package 和接口清晰分层，易于维护
- 开发速度比微服务快很多，满足 MVP 尽快交付
- 部署简单，单个 Docker 镜像
- 预留了未来扩展空间：如果后续需要，可以把分层模块拆分成独立微服务
- 符合当前阶段（内部使用，单并发）的实际需求

## Consequences

### 优点：
- ✅ 开发速度快，MVP 可以在预期时间交付
- ✅ 模块逻辑清晰分层，易于维护
- ✅ 部署简单，满足 Docker Compose 要求
- ✅ 未来可以逐步拆分成微服务，不重蹈推倒重来

### 缺点：
- ⚠️ 水平扩展只能整体扩展，不能单独扩展某个模块
- ⚠️ 全系统必须使用同一技术栈
- ⚠️ 如果真需要马上支持 1000+ 并发，这个方案不合适，但目前不需要

## Alternatives Considered

| 方案 | 优点 | 缺点 |
|---|---|---|
| 纯单体 | 开发最快 | 耦合严重，难扩展 |
| 微服务 | 最灵活最可扩展 | 开发慢，复杂度高，过度设计 |
| **分层单体** | **平衡，满足现在和未来** | **整体扩展，单一技术栈**（我们接受）|
