# Agent 验证平台 - 需求澄清

## 1. 项目概述

**项目名称**：Agent 验证平台

**业务目标**：为 AaaS/MaaS 平台构建标准化 Agent 验证基准测试平台，统一验证 Agent 的性能、正确性、稳定性，提供可对比的基准数据。

**路线图**：
- 第一期：支持内部验证 Agent（重点是 openclaw）
- 后续：对接 MaaS 对外提供验证服务

## 2. 核心功能范围

### Must-have（第一期必须实现）

| 功能模块 | 说明 |
|---|---|
| **Agent 注册管理** | 通过 `baseURL + APIKey` 方式添加 HTTP API 类型 Agent，支持编辑/删除 |
| **LLM 模型配置管理** | 配置用于结果比对的 LLM 模型，支持测试连接 |
| **测试场景管理** | 管理自定义测试场景和测试用例 |
| **用例执行** | 驱动 Agent 运行测试场景 |
| **全链路观测** | Trace/Span 数据 **从 Opik/Langfuse 的 ClickHouse 数据库获取**，不需要自己采集存储原始追踪数据 |
| **数据存储** | 存储测试结果、比对结论、指标统计 |
| **结果判定** | 与基准数据比对，判定测试结果 |
| **全链路回放** | 支持回放整个执行过程 |
| **模型比对** | 比对过程输出和最终结果（使用 LLM 做语义一致性比对） |
| **报告展示** | 展示测试结果和指标（Runtime/CPU/Memory/LLM Token 等） |

### 第一期不实现（后续迭代）

- 内置 benchmark 测试集（GAIA/BFCL/SWE-bench 等）
- CI/CD 流水线集成自动触发
- MaaS 平台对接对外服务
- 多租户/权限隔离
- 1000+ 并发压测支持

## 3. 用户角色与工作流

### 主要用户

- **测试工程师**：执行验证测试
- **产品经理**：查看测试报告和对比结果
- **管理员**：配置 LLM 比对模型、ClickHouse 连接

### 典型工作流

```
用户新增 LLM 比对模型（配置 API/测试连接）
    ↓
用户新增 Agent（baseURL + APIKey，测试连接）
    ↓
添加/选择 Agent 测试场景（含基准数据）
    ↓
执行场景用例（驱动 Agent 运行）
    ↓
从 Opik/Langfuse ClickHouse 获取 Trace/Span 数据
    ↓
自动与场景基准数据比对（结果 + 指标，使用配置的 LLM 做语义比对）
    ↓
用户查看测试报告和详情，支持全链路回放
```

## 4. 技术要求

### Agent 接入方式

- 第一期：**HTTP API 方式**（baseURL + APIKey 认证）
- 后续扩展：支持本地 SDK 调用、CLI/TUI 接入
- 需要同时支持：远程 Agent 服务（HTTP/gRPC）+ 本地部署 Agent

### LLM 比对模型

- 需要支持多模型配置管理
- 支持测试连接验证可用性
- 需要设置默认比对模型

### 数据存储

- **Trace/Span 原始数据**：从 Opik/Langfuse 的 ClickHouse 获取，平台不重复存储
- **测试结果、指标统计、比对结论**：平台自建存储
- 需要**时序数据库（Prometheus/InfluxDB）**存储聚合指标
- 测试结果保留 **1 个月**，自动清理过期数据

### 可观测性集成

- 从 Opik **或** Langfuse 的 ClickHouse 数据库读取 Trace/Span 数据
- 需要计算提取各种指标：
  - TTFT (Time to First Token)：首包延迟
  - TPOT (Time per Output Token)：每 Token 延迟
  - Token 统计（输入/输出/总量）
  - CPU 使用率、内存占用
  - 工具调用次数、错误率、延迟

### 扩展性

- 第一期不考虑多租户/权限隔离，但架构需要支持后续扩展
- 第一期不考虑高并发，架构需要支持后续扩展到 **1000+ 并发**

## 5. 非功能性需求

| 维度 | 要求 |
|---|---|
| **并发规模** | 一期：单任务执行；二期：1000+ 并发 |
| **数据保留** | 测试结果保留 1 个月 |
| **部署** | 一期：本地服务器 Docker Compose；二期：私有 Kubernetes 集群 |
