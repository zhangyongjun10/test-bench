# 版本差异测试分析技能

## 概述

本技能用于分析软件版本间的差异，智能识别代码、配置、接口、数据库等变更，自动生成测试影响分析报告和测试策略建议。适用于微服务架构、单体应用、前后端分离等各种架构下的版本升级、功能迭代和缺陷修复场景。

## 分析模式

### 模式1：双版本目录对比（推荐用于完整版本对比）

**适用场景**：比较两个完整的版本代码目录

**输入要求**：
- 旧版本代码路径：`/path/to/version-old`
- 新版本代码路径：`/path/to/version-new`

**分析内容**：
- 文件级变更（新增、删除、修改）
- 代码级变更（函数、类、接口的变化）
- 配置变更（配置项的增删改）
- 依赖变更（第三方库版本变化）

---

### 模式2：Git Diff分析（推荐用于基于版本控制的分析）

**适用场景**：基于Git仓库的commit、branch或tag进行分析

**输入要求**：
- Git仓库路径：`/path/to/git/repo`
- 对比基准：commit hash / branch name / tag name

**分析内容**：
- Commit级别的变更统计
- 文件变更详情（新增、修改、删除）
- 代码行级变更（+/-行统计）
- 变更作者和时间分布

---

### 模式3：单版本分析（推荐用于新版本首次分析）

**适用场景**：分析单个版本的代码结构，识别关键模块和测试重点

**输入要求**：
- 版本代码路径：`/path/to/version`

**分析内容**：
- 项目结构分析（模块划分、依赖关系）
- 核心功能识别（入口、主流程）
- 接口清单（API、RPC、消息队列）
- 数据模型分析（数据库、缓存）
- 配置项梳理（环境配置、业务配置）

---

## 核心分析维度

### 维度1：功能变更分析

**分析内容**：
- 新增功能模块识别
- 现有功能修改点分析
- 删除功能影响评估
- 功能依赖关系变更

**输出**：
- 功能变更清单（新增/修改/删除）
- 功能影响范围图
- 回归测试建议

---

### 维度2：接口变更分析

**分析内容**：
- API接口变更（URL、Method、参数、返回值）
- RPC接口变更（服务名、方法名、参数类型）
- 消息队列变更（Topic、Queue、消息格式）
- 事件/回调变更（事件名、Payload结构）

**输出**：
- 接口变更清单（兼容/不兼容）
- 接口依赖影响分析
- 契约测试建议

---

### 维度3：数据库变更分析

**分析内容**：
- 表结构变更（新增表、删除表、字段变更）
- 索引变更（新增、删除、修改）
- 约束变更（主键、外键、唯一约束）
- 数据迁移脚本分析
- SQL语句变更（性能影响）

**输出**：
- 数据库变更清单
- 数据迁移风险评估
- 数据一致性测试建议

---

### 维度4：配置变更分析

**分析内容**：
- 环境配置变更（开发、测试、生产环境）
- 业务配置变更（开关、阈值、参数）
- 安全配置变更（认证、授权、加密）
- 第三方服务配置变更（URL、密钥、Token）

**输出**：
- 配置变更清单
- 配置影响范围分析
- 配置验证测试建议

---

### 维度5：性能影响分析

**分析内容**：
- 算法复杂度变更（时间/空间复杂度）
- 数据库查询变更（SQL优化/劣化）
- 缓存策略变更（命中/失效策略）
- 并发处理变更（锁机制、线程池）
- 资源使用变更（内存、CPU、IO）

**输出**：
- 性能影响评估（高/中/低）
- 关键性能指标建议（QPS、RT、资源使用率）
- 性能测试策略建议（压测、负载、稳定性）

---

### 维度6：风险评估与测试策略

**分析内容**：
- 变更风险等级评估（高/中/低）
- 影响范围评估（模块、功能、用户）
- 兼容性风险评估（向前/向后兼容）
- 回滚风险评估（数据、版本、配置）

**输出**：
- 综合风险评估报告
- 测试优先级建议（P0/P1/P2/P3）
- 回归测试范围建议
- 专项测试建议（安全、性能、兼容）
- 上线检查清单

3. **接口兼容性测试** → 验证接口兼容性
4. **数据库迁移测试** → 验证数据库变更
5. **回归测试** → 验证存量功能

| 中期 | 全面覆盖，深入测试 | 发现功能缺陷 |

## 使用流程

### 步骤1：确定分析模式

根据提供的输入，自动判断适合的分析模式：

| 输入情况 | 建议模式 | 说明 |
|---------|---------|------|
| 提供两个版本目录 | 双版本目录对比 | 最完整，推荐 |
| 提供Git仓库和commit/tag | Git Diff分析 | 适合基于Git管理的项目 |
| 只提供单个版本 | 单版本分析 | 首次分析或无法获取旧版本时使用 |

---

### 步骤2：执行差异分析

根据选定的模式，执行以下分析：

1. **文件变更分析**
   - 识别新增、删除、修改的文件
   - 按文件类型分类统计
   - 标记关键配置文件变更

2. **代码变更分析**
   - 识别函数/方法级变更
   - 分析代码逻辑变化
   - 识别核心业务逻辑变更

3. **配置变更分析**
   - 分析配置文件变更
   - 识别关键配置项变化
   - 评估配置变更影响

4. **依赖变更分析**
   - 识别第三方库版本变更
   - 分析依赖冲突风险
   - 评估依赖升级影响

---

### 步骤3：生成测试影响分析

基于差异分析结果，生成测试影响分析报告：

1. **变更影响评估**
   - 功能影响范围评估
   - 接口兼容性评估
   - 数据兼容性评估
   - 性能影响评估

2. **风险等级评估**
   - 变更复杂度评估
   - 测试难度评估
   - 上线风险评估
   - 回滚风险评估

3. **测试策略建议**
   - 测试重点建议
   - 测试优先级建议
   - 回归测试范围建议
   - 专项测试建议

---

### 步骤4：输出分析报告

按照规定的格式输出完整的测试影响分析报告。

---

## 输出要求

### 必须输出的内容

1. **分析概览**：分析模式、对象、时间、范围等基本信息
2. **变更统计**：文件级和代码级的变更统计数据
3. **详细变更分析**：
   - 功能变更分析（新增/修改/删除）
   - 接口变更分析（API/RPC/消息）
   - 数据库变更分析（表/字段/索引）
   - 配置变更分析（环境/业务/安全）
   - 性能影响分析（算法/查询/缓存）
4. **风险评估**：风险等级分布、关键风险详述
5. **测试策略建议**：
   - 测试优先级矩阵
   - 测试范围建议（必须/建议/可选）
   - 专项测试建议（接口/数据库/性能）
   - 测试数据准备建议
   - 测试环境建议
   - 测试执行建议
6. **上线检查清单**：功能/性能/质量/准备清单
7. **附录**：常见变更类型、风险评估矩阵、优先级定义、参考模板

### 输出格式要求

- 输出格式：Markdown
- 必须包含所有章节，如某维度无变更，标注"本次无变更"
- 表格内容必须完整，不得省略关键信息
- 风险分析不少于4条
- 所有测试相关的技术术语需添加中文翻译（如 API → API (应用程序接口)）

### 输出质量要求

- 分析结论必须有数据支撑，不得凭空想象
- 风险评估必须客观，不得夸大或缩小
- 测试建议必须具体可行，不得泛泛而谈
- 必须考虑各种边界情况和异常场景

## 使用示例

### 示例1：Java服务版本升级分析

**输入**：

旧版本：/path/to/service-v1.0.0

**分析重点**：
- Java源码变更（.java文件）

- Maven依赖变更（pom.xml）
- 数据库脚本变更（.sql）

**输出**：完整的测试影响分析报告

**分析重点**：

**分析重点**：

- 文件变更详情
- 代码行级变更

**输出**：完整的测试影响分析报告

## 注意事项

1. **输入路径检查**：确保提供的代码目录或Git仓库路径存在且可访问
2. **权限检查**：确保有足够的权限读取代码文件和执行Git命令
3. **大仓库优化**：对于大型代码仓库，分析可能需要较长时间，请耐心等待
4. **二进制文件**：本技能主要针对文本文件，二进制文件只会被标记为变更，不会进行内容分析
5. **敏感信息**：分析过程中可能会接触到敏感信息（如密码、密钥），请确保遵守安全规范
