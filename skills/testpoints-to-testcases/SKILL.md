name: testpoints-to-testcases
description: 测试点转测试用例。基于需求文档和测试点清单，将抽象的测试点转换为详细、可执行的测试用例。支持编写具体的测试步骤、预期结果、前置条件、用例等级等完整信息。支持多种输出格式：Markdown表格、JSON、Excel可导入CSV。适用场景：已获取测试点清单，需要将测试点具体化为可执行的测试用例时使用
---

## 技能依赖

本技能依赖 `requirements-to-testpoints` 技能生成的测试点清单，工作流衔接：
1. `requirements-to-testpoints` → 提取测试点清单（输出 `testpoints.md`）
2. `testpoints-to-testcases` → 测试点转可执行测试用例（输出 `testcases.*`）

## 参考文档说明

本 skill 的 `references/` 目录下包含以下参考文档，**生成测试用例时必须读取并基于这些文档**：

| 文件名 | 内容说明 | 使用场景 | 必填 |
|--------|---------|---------|------|
| `需求分析.md` | 业务需求的功能分析文档 | 理解业务功能、操作流程、界面交互的核心依据 | ✅ |
| `详细设计.md` | 系统的技术设计文档 | 补充数据规则、字段约束、逻辑细节 | ⭕ 可选 |
| `需求澄清.md` | 需求不明确时的澄清记录 | 确认边界条件和特殊规则 | ⭕ 可选 |
| `testpoints.md` | 需求分析师提供的测试点清单 | **测试点的唯一来源**，用例数量和顺序以此为准 | ✅ |

## 核心工作流程（必须严格遵循）

| 步骤 | 动作 | 关键要点 | 强制 |
|-----|------|---------|------|
| **Step 1** | 读取参考文档 | **必须先读取** `references/` 下的所有参考文档 | ✅ |
| **Step 2** | 解析测试点清单 | 从 `testpoints.md` 中**完整提取所有测试点**，按模块分类统计 | ✅ |
| **Step 3** | 理解业务细节 | 基于参考文档深入理解业务功能、操作流程、界面交互、数据规则等细节 | ✅ |
| **Step 4** | 输出测试点统计 | 按模块统计测试点数量，输出统计结果给用户确认 | ✅ |
| **Step 5** | 按顺序编写用例 | 每个测试点对应一个测试用例，**严格按测试点顺序编写，不得跳过、不得调整顺序** | ✅ |
| **Step 6** | 编写测试步骤 | 测试步骤必须**来源于需求文档的实际功能描述**，不能仅凭测试点名称想象 | ✅ |
| **Step 7** | 最终一致性检查 | 确保**测试用例数量 = 测试点数量**，**编号顺序完全一致** | ✅ |

## 核心设计原则（必须严格遵守）

### 基本原则
- **需求分析师的测试点 = 用例标题和测试范围指导**：用例名称必须与测试点完全一致，不得修改
- **需求文档的功能描述 = 测试步骤的具体内容来源**：所有步骤必须能在参考文档中找到依据
- **绝不能**仅仅基于测试点名称就编写测试步骤，必须回到需求文档找到对应的功能细节
- **一个测试点 → 一个测试用例**：每个测试点都必须有对应的测试用例，不能遗漏任何一个
- **编号一致性**：测试点编号直接作为测试用例编号，保持一致便于追溯

### 编写规范
- **步骤具体可操作**：每个测试步骤都要包含具体的操作描述（点击什么按钮、输入什么数据、在哪个页面等）
- **预期结果可验证**：明确说明期望看到什么结果、什么状态变化、什么数据展示，不能模糊
- **前置条件清晰**：准确描述执行测试前必须满足的环境、数据、权限、状态等条件

## 特殊情况处理

| 场景 | 处理方式 |
|------|---------|
| **需求文档中找不到对应细节** | 在备注字段说明：`注：需求文档未明确描述此功能，步骤基于合理假设`，同时在输出开头整体标注 |
| **需求描述模糊/不完整** | 在输出开头标注「**部分需求描述不完整，以下测试用例基于合理假设**」，并列出假设前提和疑问点 |
| **测试点名称已经包含完整说明** | 保留完整测试点名称作为用例名称，不要截断修改 |

| API调用 | `接口测试` |

## 用例等级划分规则（必须严格遵循）

**强制要求：任何项目输出必须包含 P0 级别测试用例**，必须识别并标记主干冒烟核心入口测试点为 P0。

自动按以下规则划分用例等级：

| 等级 | 适用场景 | 发布要求 |
|------|---------|---------|
| **P0** | 主干冒烟用例、核心主流程入口 → 只有满足以下条件才划分为P0：<br>1. 产品主干流程第一个入口（登录、注册、首页打开、主功能进入）<br>2. 不测试直接影响整个版本验证的核心入口<br>3. 整个测试团队冒烟测试必须跑的用例 | 版本发布必须100%通过，不通过直接block |
| **P1** | 阻塞主流程、核心业务功能、涉及资金权限安全 → 核心业务功能的正常流程测试点都划分为P1 | 必须测试，发布前必须通过 |
| **P2** | 重要业务功能、非核心边界异常 → 边界值、异常场景都划分为P2 | 需要测试 |
| **P3** | 次要功能、UI细节、异常提示 → 次要功能、非关键场景 | 可选测试 |
| **P4** | 极其次要、不影响用户使用 → 极边缘场景 | 可延后测试 |

## 输出格式选择

用户可以指定输出格式，如果未指定，默认输出JSON。支持三种格式：

### 1. JSON格式（默认，推荐用于导入测试管理系统）

```json
{
  "业务模块名称": [
    {
      "ID": "TC-XXX",
      "用例名称": "[测试点名称]（直接使用需求分析师的测试点名称，不要修改）",
      "所属模块": "业务模块名称",
      "前置条件": "前置条件描述（基于需求文档的具体要求）",
      "备注": "测试用例相关备注说明（没有则留空）",
      "步骤描述": "具体操作步骤1（基于需求文档的功能描述）\n具体操作步骤2（基于需求文档的功能描述）\n具体操作步骤3（基于需求文档的功能描述）",
      "预期结果": "具体预期结果1（基于需求文档的功能要求）\n具体预期结果2（基于需求文档的功能要求）\n具体预期结果3（基于需求文档的功能要求）",
      "编辑模式": "创建",
      "标签": "自动生成的标签，遵循标签规则",
      "用例等级": "P0/P1/P2/P3/P4",
      "用例状态": "待执行"
    }
  ]
}
```

### 2. Markdown表格格式（推荐用于文档评审）

| ID | 用例名称 | 所属模块 | 前置条件 | 步骤描述 | 预期结果 | 用例等级 |
|----|----------|----------|----------|----------|----------|----------|
| TC-001 | 完整信息注册 | 用户注册 | 用户打开注册页面 | 1. 输入用户名... 2. 点击注册 | 1. 注册成功... | P1 |

### 3. CSV格式（可直接导入Excel/测试管理工具）

输出标准CSV格式，包含表头：
```
ID,用例名称,所属模块,前置条件,步骤描述,预期结果,用例等级,标签,用例状态
```

生成后可以直接在Excel中打开，或导入到TestLink/Jira/ZenTao等测试管理系统。

## 质量检查清单（输出前必须完成自检）

在输出最终结果前，请进行以下自检：

✅ **数量一致性**：测试用例数量是否等于测试点数量？无遗漏无多增？  
✅ **顺序一致性**：测试用例顺序和编号是否与测试点完全一致？  
✅ **内容依据**：所有测试步骤和预期结果是否都能在参考文档中找到依据？  
✅ **步骤可操作**：测试步骤是否具体清晰？是否说明了在哪个页面点击什么按钮输入什么内容？  
✅ **结果可验证**：预期结果是否明确可验证？是否说明了应该看到什么？  
✅ **等级正确性**：用例等级划分是否符合等级划分规则？  
✅ **P0 要求满足**：是否识别并输出了 P0 级别冒烟测试用例？是否满足强制要求必须包含 P0？  
✅ **标签正确性**：标签是否按照测试点类型正确生成？  
✅ **格式正确性**：输出格式是否符合用户要求的格式规范？  
✅ **没有臆造**：是否存在仅凭测试点名称臆造步骤？是否添加了需求中没有的功能？  
✅ **特殊情况处理**：需求不完整时是否标注了假设和疑问点？

## 使用说明

- 如果用户**未指定格式**，默认输出**JSON格式**
- 如果用户要求**Excel**，输出**CSV格式**（CSV可以直接在Excel中打开，也可导入大多数测试管理系统）
- 如果用户需要**文档评审**，输出**Markdown表格格式**
- 输出文件必须保存到 skill 当前工作目录，文件名固定为 `testcases.${format}`
