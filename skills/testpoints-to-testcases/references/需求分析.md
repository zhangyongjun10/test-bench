# Agent 验证平台 - 需求分析

## 1. 需求概览

本项目是为 AaaS/MaaS 平台构建的标准化 Agent 验证平台，支持对 HTTP API 接入的 Agent 进行自定义场景验证，从 Opik/Langfuse 获取全链路追踪数据，使用 LLM 进行结果语义比对，输出性能指标分析报告，支持全链路执行回放。

## 2. 用户故事分解

### 📦 Epic 1: Agent 生命周期管理

**Feature 1.1: Agent 创建与注册**

```
As a 测试工程师
I want 通过 baseURL + APIKey 方式添加一个 HTTP Agent
So that 我可以对这个 Agent 进行验证测试

Acceptance Criteria:
- [ ] 提供表单输入 Agent 名称、描述、baseURL、APIKey
- [ ] APIKey 敏感信息加密存储
- [ ] 支持测试连接，验证 Agent 是否可正常调用
- [ ] 保存 Agent 配置到数据库
```

```
As a 测试工程师
I want 编辑已注册的 Agent 信息
So that 我可以修改 Agent 连接配置

Acceptance Criteria:
- [ ] 可以编辑所有字段
- [ ] 可以重新测试连接
- [ ] 修改后保存更新
```

```
As a 测试工程师
I want 删除不需要的 Agent
So that 保持列表整洁

Acceptance Criteria:
- [ ] 删除前有二次确认
- [ ] 删除后同时清理关联的测试结果
```

**Feature 1.2: Agent 查询与列表**

```

I want 查看所有已注册 Agent 列表
So that 我可以选择要测试的 Agent
```

### 🔍 Epic 5: 可观测性数据集成

**Feature 5.1: ClickHouse 连接配置**

```
As a 管理员
I want 配置 Opik/Langfuse ClickHouse 连接信息
So that 系统可以拉取 Trace 数据

Acceptance Criteria:
- [ ] 配置连接地址、账号密码
- [ ] 支持测试连接
- [ ] 支持切换 Opik 或 Langfuse 数据源
```

**Feature 5.2: 指标提取与计算**

```
As a 系统
I want 从 Span 数据提取各类性能指标
So that 可以进行性能分析比对

Acceptance Criteria:
- [ ] 提取 LLM 指标：TTFT（首包延迟）、TPOT（每 token 延迟）、Token 数量统计
- [ ] 提取 Runtime 指标：CPU 使用率、内存占用、GC 次数/耗时
- [ ] 提取工具调用指标：调用次数、错误率、平均延迟
- [ ] 计算百分位数：P50/P75/P90/P99
- [ ] 存储所有指标到时序数据库
```

### ⚖️ Epic 6: 结果比对与分析

**Feature 6.1: 结果比对**

```
As a 系统
I want 将测试结果与基准数据进行比对
So that 可以自动判定测试是否通过

Acceptance Criteria:
- [ ] 结果内容比对：使用 LLM 比对输出语义一致性
- [ ] 过程比对：比对调用链路、工具调用顺序是否符合预期
- [ ] 指标比对：对比性能指标（TTFT/Token 等）是否满足 SLO
- [ ] 输出比对分数和判定结果（通过/不通过）
```

**Feature 6.2: 异常检测**

```
As a 系统
I want 检测测试过程中的异常
So that 用户可以快速发现问题

Acceptance Criteria:
- [ ] 检测工具调用错误率超过阈值
- [ ] 检测延迟超过 SLO
- [ ] 检测 OOM、超时等异常
- [ ] 在报告中标记异常点
```

## 3. 依赖关系分析

### 依赖图

```
[基础设施 数据存储]
         ↓
[LLM 模型配置管理] [Agent 注册管理] [场景管理] [ClickHouse 配置]
         ↓             ↓             ↓           ↓
         └─────────────┴─────────────┴──────────────┘
                                     ↓
                             测试执行引擎
                                     ↓
                             拉取 Trace 数据
                                     ↓
                             指标提取计算
                                     ↓
                             结果比对分析
                                     ↓
                 ┌─────────────────┴─────────────────┐
                 ↓                                   ↓
               报告展示                           全链路回放
                 ↓                                   ↓
             历史对比                           数据导出
```

### 依赖类型分类

| 依赖类型 | 描述 |
|---|---|
| **强依赖** | 必须先做 A 才能做 B |
| | 基础设施 → 所有业务功能 |
| | LLM 模型配置 → 结果比对 |
| | Agent/场景/ClickHouse 配置 → 测试执行 |
| | Trace 拉取 → 指标提取 → 结果比对 → 报告 |
| **弱依赖** | 可并行开发，但需要接口约定 |
| | 前端开发 ↔ 后端开发 |
| **无依赖** | 完全独立 |
| | Docker Compose 配置（最后做）|
| | 历史对比（基础报告完成后）|

## 4. 技术可行性评估

| Epic | 技术复杂度 | 未知风险 | 风险评估 | 需要调研 |
|---|---|---|---|---|
| Agent 生命周期管理 | ⭐ 低 | 无 | 低风险 | - |
| LLM 模型配置管理 | ⭐ 低 | 无 | 低风险 | - |
| 测试场景管理 | ⭐ 低 | 无 | 低风险 | - |
| 测试执行引擎 | ⭐⭐ 中 | 异步执行、超时处理 | 中风险 | 超时控制策略 |
| ClickHouse 集成 | ⭐⭐ 中 | Opik/Langfuse 表结构不确定 | 中风险 | 需要确认 Opik/Langfuse ClickHouse schema |
| 指标提取计算 | ⭐⭐ 中 | 百分位数计算效率 | 低风险 | - |
| 结果比对（LLM 语义比对）| ⭐⭐⭐ 高 | 比对准确性依赖 LLM | 中风险 | 需要原型验证 LLM 比对准确率 |
| 全链路回放 | ⭐⭐ 中 | 大 trace 前端渲染性能 | 低风险 | - |
| 报告可视化 | ⭐⭐ 中 | 多维度指标展示 | 低风险 | - |
| 基础设施存储 | ⭐⭐ 中 | 时序数据库选型 | 低风险 | 选型比较 |
| Docker Compose 部署 | ⭐ 低 | 无 | 低风险 | - |

### 总体风险评估

- 项目整体技术可行，没有不可解决的技术难题
- 最大的不确定性在于：
  1. **Opik/Langfuse ClickHouse schema** 需要确认
  2. **LLM 语义比对** 的准确性需要验证
- 建议开工前做一个小 **spike** 验证这两点

## 6. 实施计划

### 阶段 0: Spike 前期调研 (1-2 天)

- [ ] 确认 Opik/Langfuse ClickHouse 表结构
- [ ] 验证 LLM 语义比对方案的有效性
- [ ] 确认时序数据库选型 (Prometheus vs InfluxDB)

### 阶段 1: 基础设施 (2-3 天)

- [ ] 初始化项目结构（前后端分离）
- [ ] 创建数据库表结构（关系库：Agent/场景/LLM配置/执行任务）
- [ ] 配置时序数据库
- [ ] 搭建基础开发框架（后端 API + 前端脚手架）

### 阶段 2: 基础功能 - 管理模块 (4-5 天)

- [ ] 后端 API: Agent CRUD
- [ ] 后端 API: LLM 模型 CRUD
- [ ] 后端 API: 测试场景 CRUD
- [ ] 前端: Agent 列表、创建、编辑页面
- [ ] 前端: LLM 模型列表、创建、编辑页面
- [ ] 前端: 场景列表、创建、编辑页面

### 阶段 3: 核心执行引擎 (3-5 天)

- [ ] ClickHouse 连接配置管理
- [ ] 测试执行任务框架（同步调用 + 异步处理）
- [ ] HTTP Agent 调用封装
- [ ] Trace 数据拉取和解析
- [ ] 指标提取和计算（TTFT/TPOT/CPU/Memory 等）
- [ ] 存储指标到时序数据库

### 阶段 4: 比对分析 (2-3 天)

- [ ] LLM 语义比对实现
- [ ] 指标阈值比对
- [ ] 异常检测

### 阶段 5: 回放和报告 (3-4 天)

- [ ] 全链路回放前端展示
- [ ] 测试报告页面（指标展示）
- [ ] 异常标记展示

### 阶段 6: 收尾部署 (1-2 天)

- [ ] Docker Compose 配置
- [ ] 文档整理
- [ ] 联调测试

### 总体预估

| 指标 | 预估 |
|---|---|
| **总工作量** | 16-23 人天 |
| **2 人并行开发** | 1-2 周 MVP 可交付 |

### 并行开发建议

- 后端：按阶段顺序开发
- 前端：同步跟进后端 API 开发页面
- 前后端可以完全并行，只需要提前约定 API 接口契约
