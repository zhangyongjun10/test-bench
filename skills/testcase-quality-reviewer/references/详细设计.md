## 1. 架构决策总结

| 决策 | 选择 |
|---|---|
| **整体架构** | 分层单体（模块按 package 拆分，同一进程）|
| **后端** | Python + FastAPI（异步）|
| **前端** | React + Ant Design |
| **关系数据库** | PostgreSQL |
| **时序数据库** | Prometheus |
| **ClickHouse** | 只读 Opik/Langfuse 数据，不存储 |
| **部署** | Docker + Docker Compose（一期）|
| **扩展性** | 模块接口分离，未来可拆分微服务 |

## 2. 数据模型设计

### 2.1 实体关系图

```
┌─────────────┐
│  llm_models  │
└──────┬──────┘
       │ 1
       │
       ▼
┌─────────────┐
│   agents    │
└──────┬──────┘
       │ 1
       │
       ├───────────┐
       │           │
       ▼           ▼
┌─────────────┐     │
│  scenarios  │     │ 一个 Agent 有 N 个场景
└──────┬──────┘     │
       │ 1          │
       │            │
       │            ▼
       │         ┌─────────────┐
       └────────▶│ execution_  │
                 │  jobs      │  --->  一个 job 属于一个 agent, 一个 scenario, 一个 llm
                 └─────────────┘
```

**关系说明**：
- 一个 Agent 可以有**多个场景**（1:N）
- 一个场景只能属于**一个 Agent**（每个场景绑定到特定 Agent）
- 一个执行 Job 引用一个 Agent、一个场景、一个 LLM 模型

### 2.2 数据库表结构

#### 2.2.1 `agents` - Agent 注册表

```sql
CREATE TABLE agents (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name VARCHAR(255) NOT NULL,
    description TEXT,
    base_url VARCHAR(2048) NOT NULL,
    api_key_encrypted TEXT NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    deleted_at TIMESTAMP WITH TIME ZONE
);

-- 索引
CREATE INDEX idx_agents_deleted_at ON agents(deleted_at);
CREATE INDEX idx_agents_name ON agents(name);
```

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | UUID | PK | 唯一标识 |
| name | varchar(255) | NOT NULL | Agent 名称 |
| description | text |  | 描述说明 |
| base_url | varchar(2048) | NOT NULL | Agent API 基地址 |
| api_key_encrypted | text | NOT NULL | 加密后的 API Key |
| created_at | timestamp | NOT NULL | 创建时间 |
| updated_at | timestamp | NOT NULL | 更新时间 |
| deleted_at | timestamp |  | 软删除标记 |

#### 2.2.2 `llm_models` - LLM 比对模型配置

```sql
CREATE TABLE llm_models (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
```

#### 2.2.3 `scenarios` - 测试场景

```sql
CREATE TABLE scenarios (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    agent_id UUID NOT NULL REFERENCES agents(id),
    name VARCHAR(255) NOT NULL,
    description TEXT,
    prompt TEXT NOT NULL,
    baseline_result TEXT,
    compare_result BOOLEAN NOT NULL DEFAULT TRUE,
    compare_process BOOLEAN NOT NULL DEFAULT FALSE,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    deleted_at TIMESTAMP WITH TIME ZONE
);

-- 索引
CREATE INDEX idx_scenarios_deleted_at ON scenarios(deleted_at);
CREATE INDEX idx_scenarios_agent_id ON scenarios(agent_id);
CREATE INDEX idx_scenarios_name ON scenarios(name);
```

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | UUID | PK | 唯一标识 |
| **agent_id** | UUID | NOT NULL, FK | 关联的 Agent（每个场景属于特定 Agent） |
| name | varchar(255) | NOT NULL | 场景名称 |
| description | text |  | 描述说明 |
| prompt | text | NOT NULL | 基准问题/提示词 |
| baseline_result | text |  | 基准预期结果 |
| compare_result | boolean | 默认 true | 是否比对结果 |

| created_at | timestamp | NOT NULL | 创建时间 |
| updated_at | timestamp | NOT NULL | 更新时间 |
| deleted_at | timestamp |  | 软删除标记 |

#### 2.2.4 `execution_jobs` - 执行任务

```sql
CREATE TABLE execution_jobs (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    agent_id UUID NOT NULL REFERENCES agents(id),
    scenario_id UUID NOT NULL REFERENCES scenarios(id),
    llm_model_id UUID REFERENCES llm_models(id),
    trace_id VARCHAR(255),
    status VARCHAR(50) NOT NULL,
    original_request TEXT,
    original_response TEXT,
    error_message TEXT,
    comparison_score DOUBLE PRECISION,
    comparison_passed BOOLEAN,
    started_at TIMESTAMP WITH TIME ZONE,
    completed_at TIMESTAMP WITH TIME ZONE,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
);

-- 索引
CREATE INDEX idx_execution_jobs_agent_id ON execution_jobs(agent_id);
CREATE INDEX idx_execution_jobs_scenario_id ON execution_jobs(scenario_id);
CREATE INDEX idx_execution_jobs_status ON execution_jobs(status);
CREATE INDEX idx_execution_jobs_created_at ON execution_jobs(created_at DESC);
```

**状态枚举**：`queued` / `running` / `pulling_trace` / `comparing` / `completed` / `failed`

## 5. 核心业务逻辑

### 5.1 执行测试场景流程

```
用户触发 POST /api/v1/execution
         ↓
1. 校验参数：agent_id/scenario_id 是否存在
         ↓
2. 创建 execution_jobs 记录，status = queued
         ↓
3. 后台任务（BackgroundTasks）添加异步执行：run_execution(execution_id)
         ↓
4. 返回 200，给前端：execution_id, status=queued
         ↓
─────────────────────────────────────────────────
         后台异步执行
─────────────────────────────────────────────────
         ↓
A. 更新 status = running，记录 started_at = now()
         ↓
B. 获取 Agent 配置 → HTTP Agent 客户端调用
         ↓
C. 保存 original_request / original_response 到数据库
   如果调用失败 → 更新 status=failed，记录 error_message，结束
         ↓
D. 更新 status = pulling_trace
         ↓
E. trace_id 由平台生成，在 HTTP 请求 Header 传给 Agent
         ↓
F. 调用 TraceFetcher.fetch_spans(trace_id) 从 ClickHouse 查询
         ↓
G. MetricExtractor.extract(spans) 提取各类指标
         ↓
H. 存储指标到 Prometheus
         ↓
I. 更新 status = comparing
         ↓
J. Comparator.compare() → 语义比对 + 指标比对 → score + passed + 异常列表
         ↓
K. 更新 execution 记录：score, passed, completed_at = now()
         ↓
L. 更新 status = completed
         ↓
M. 结束
```

### 5.2 指标提取算法

```python
def extract(spans: List[Span]) -> ExecutionMetrics:
    llm_spans = [s for s in spans if s.span_type == "llm"]
    tool_spans = [s for s in spans if s.span_type == "tool"]
    runtime_spans = [s for s in spans if s.span_type == "runtime"]

    # 提取 LLM 指标
    ttft_list = [s.metrics.ttft_ms for s in llm_spans if s.metrics.ttft_ms]
    tpot_list = [s.metrics.tpot_ms for s in llm_spans if s.metrics.tpot_ms]
    input_tokens = sum(s.metrics.input_tokens for s in llm_spans)
    output_tokens = sum(s.metrics.output_tokens for s in llm_spans)

    # 计算百分位数
    llm_metrics = LLMMetrics(
        ttft_ms=percentile(ttft_list, [50, 75, 90, 99]),
        tpot_ms=percentile(tpot_list, [50, 75, 90, 99]),
        total_tokens=input_tokens + output_tokens,
        input_tokens=input_tokens,
        output_tokens=output_tokens
    )

    # ... 同理处理 runtime 和 tool 指标

    return ExecutionMetrics(
        llm=llm_metrics,
        runtime=runtime_metrics,
        tool=tool_metrics
    )
```

**百分位数**：使用 `numpy.percentile`

### 5.3 LLM 语义比对

**Prompt 模板**：

```
你是一个 AI 输出比对专家。请比对实际输出和预期结果的语义一致性。

问题：
{question}

预期结果：
{baseline}

实际输出：
{actual}

请判断：
1. 实际输出和预期结果在语义上是否一致？是否满足问题要求？
2. 给一个一致性分数 0 到 1，0 表示完全不一致，1 表示完全一致。

输出 JSON 格式：
{
  "consistent": true/false,
  "score": 0.x,
  "reason": "简要解释为什么这么打分"
}
```

### 5.4 状态转换

```

     ┌─────────┐
     │ running │  ←  开始调用 Agent
     └───┬─────┘

     └───┬──────┘
         │ 比对完成
         ▼
    ┌─────────┐    ┌────────┐
    │completed│    │ failed │
    └─────────┘    └────────┘
```

### 5.5 数据自动清理

- 应用启动时，启动后台 scheduler
- 每天凌晨运行一次
- 删除 `created_at < now() - 30 days` 的 execution_jobs
- Prometheus 配置 retention 30 天，自动清理

## 6. 错误处理与监控

### 6.1 错误分类处理

| 错误类型 | 处理策略 | 返回 |
|---|---|---|
| 参数错误 | FastAPI 自动校验 | 400 + 错误信息 |
| 资源不存在 | 记录日志 | 404 + message |
| 业务错误 | 业务判断 | 200 code=1 + message |
| 调用 Agent 失败 | 记录 error，更新 job 状态 failed | 后台完成，前端看状态 |
| ClickHouse 连接失败 | 测试连接返回错误，执行 job failed | 前端可见 |
| 系统错误 | 记录 critical 日志 | 500 + 通用错误信息 |

### 6.2 边界条件

| 边界 | 处理方式 |
|---|---|
| 空列表 | 返回空数组 |
| 分页超出范围 | 返回空数组 |
| 超时（Agent 调用） | 默认 300 秒超时，标记 failed |
| APIKey 存储 | 加密存储，API 不返回 |

### 6.3 日志策略

| 级别 | 事件 |
|---|---|
| `INFO` | 执行开始/完成，创建/更新/删除资源 |
| `WARNING` | 比对分数低，指标超过阈值 |
| `ERROR` | 执行失败，调用失败，LLM 比对失败 |
| `DEBUG` | 详细 trace 数据，指标计算过程 |

### 6.4 监控指标

```
# 执行计数
testbench_executions_total{status="completed"} 1
testbench_executions_total{status="failed"} 1

# 执行耗时分布

# 比对分数分布
testbench_comparison_score histogram

# ClickHouse 查询耗时
testbench_clickhouse_query_duration_seconds histogram

# LLM 比对耗时
testbench_llm_compare_duration_seconds histogram
```
